# 9.2 Reflexão Crítica sobre o Uso de LLMs

## Papel dos LLMs no SGCLC

A integração de LLMs neste projecto foi genuinamente transversal a todas as etapas de desenvolvimento — não se resumiu a um momento isolado de "consulta ao ChatGPT". Esta secção apresenta uma reflexão honesta e crítica sobre o que os LLMs fizeram bem, o que fizeram mal, e o que a equipa aprendeu sobre a sua utilização em contexto de engenharia de software.

## O Que Funcionou Bien

### Eliciação de Requisitos (Etapa 1)

A utilização de LLMs para simular entrevistas com personas foi o ponto de maior valor observado. O LLM, quando adequadamente configurado com o contexto da QuickMart e o perfil do stakeholder, gerou respostas plausíveis que revelaram requisitos implícitos que as reuniões presenciais não tinham produzido (ex.: a preferência por tabela de produtos vs. pesquisa simples no POS; o pedido de log de auditoria de ajustes de stock).

O refinamento de requisitos ambíguos também foi valioso: em 8 dos 42 RFs, o LLM identificou lacunas e sugeriu formulações mais precisas e testáveis. As versões refinadas foram directamente incorporadas no SRS após validação da equipa.

### Elaboração de Documentação (Etapa 1 e 2)

O LLM acelerou significativamente a produção de documentação técnica — desde a estrutura do SRS até à redacção das user stories em formato padronizado. Sem LLM, o tempo estimado para produzir o SRS seria 2–3 vezes superior.

### Sugestão de Padrões de Design (Etapa 2)

Quando a equipa questionou o LLM sobre a melhor forma de implementar a consolidação automática em Blazor Server, o modelo sugeriu correctamente a utilização de `IHostedService` / `BackgroundService` — um padrão que alguns membros da equipa não conheciam antes deste projecto.

## O Que Não Funcionou Como Esperado

### Geração de Código (Etapa 3)

O código gerado pelo LLM para componentes Blazor complexos (ex.: POS com filtro reactivo) foi frequentemente incorrecto ou desactualizado. Partes do código gerado utilizavam APIs depreciadas de versões anteriores do .NET ou ignoravam restrições específicas do Blazor Server (ex.: limitações de ciclo de vida dos componentes). Em todos os casos, o código teve de ser revisto e corrigido pela equipa antes de poder ser utilizado.

**Lição:** O LLM é útil como ponto de partida para código boilerplate, mas não é fiável para cógido específico de framework sem revisão humana.

### Verificação de Requisitos (Etapa 1)

Em dois casos, o LLM sugeriu requisitos que pareciam plausíveis mas eram irrelevantes para o contexto da QuickMart (ex.: sugestão de um sistema de pontos de fidelização para clientes — fora do âmbito). O cliente rejeitou estas sugestões na reunião de validação.

**Lição:** O LLM optimiza para plausibilidade, não para adequação ao contexto específico. A validação com stakeholders reais é insubstituível.

### Estimativas e Dados Quantitativos

Quando questionado sobre estimativas de custo, tempo ou volume de dados, o LLM produziu valores que eram plausíveis mas não verificáveis. A equipa optou por nunca utilizar estimativas geradas pelo LLM sem validação externa.

## Balanço Final

| Dimensão | Benefício Real | Risco Real |
|---|---|---|
| Produtividade em documentação | Alto (2–3× mais rápido) | Baixo (documentação validável) |
| Descoberta de requisitos | Médio (revelou requisitos implícitos) | Médio (gerou requisitos irrelevantes) |
| Qualidade de código | Baixo (código frequentemente errado) | Alto (código errado dificilmente detectável) |
| Aprendizagem de novas tecnologias | Alto (sugestão de padrões e APIs) | Médio (APIs podem estar desactualizadas) |

**Conclusão:** Os LLMs mostraram ser mais valiosos nas fases de análise e documentação do que na implementação. A atitude correcta é tratar o LLM como um *júnior entusiasta*: produtivo, criativo, mas necessitando sempre de supervisão e revisão crítica por parte de um engenheiro experiente.

A obrigatoriedade de uso de LLMs imposta pela UC LI4 revelar-se-á uma preparação valiosa para um contexto profissional em que estas ferramentas são cada vez mais prevalentes — mas onde o julgamento humano continua a ser a competência diferenciadora.
