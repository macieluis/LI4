# 9.3 Limitações, Trabalho Futuro e Propostas de Evolução

## Limitações da Versão 1.0

O SGCLC v1.0 é uma versão funcional que resolve os problemas identificados no diagnóstico inicial. No entanto, existem limitações conhecidas que decorrem de restrições de tempo da UC LI4 ou de decisões deliberadas de escopo:

### Limitações de Funcionalidade

| Limitação | Impacto | RF Associado |
|---|---|---|
| Exportação de faturas para PDF não implementada | Utilizadores não podem imprimir faturas formatadas | RF30 |
| Rastreamento de validades de produtos não implementado | Sem alertas para produtos próximos do prazo | RF22 |
| Exportação de relatórios para CSV não implementada | Análise de dados em ferramentas externas mais difícil | RF42 |
| Alerta de código de produto duplicado em tempo real ausente | Erro só detectado ao submeter o formulário | RF10 |
| Sem paginação nas listagens | Listas podem tornar-se lentas com centenas de registos | RNF01 (parcial) |

### Limitações Técnicas

| Limitação | Impacto | Mitigação em v1.1 |
|---|---|---|
| Base de dados SQLite em desenvolvimento | Sem suporte a concorrência elevada | Migrar para SQL Server com connection string |
| Sem integração com AT (Autoridade Tributária) | Faturas não comunicadas fiscalmente | Implementar API SAFT-PT ou comunicação AT |
| Sem sistema de notificações push | Alertas visíveis apenas ao carregar a página | WebSockets ou SignalR events |
| Logging com Console.WriteLine em alguns serviços | Logs não estruturados, difíceis de monitorizar | Migrar para ILogger com Serilog |
| Sem testes de integração | Repositórios não testados automaticamente | xUnit + EF Core In-Memory provider |

### Limitações de Escalabilidade

O Blazor Server mantém um circuito SignalR por utilizador no servidor. Para até 50 utilizadores simultâneos (realidade da QuickMart), o desempenho é adequado. Para escenários com centenas de utilizadores concorrentes, seria necessário considerar:
- Migration para Blazor WebAssembly com API REST separada
- Ou deployment em múltiplas instâncias com session affinity

## Propostas de Evolução — Versão 1.1 (Curto Prazo)

As seguintes funcionalidades estão priorizadas para a v1.1, com base na sua importância para o negócio:

| Funcionalidade | Impacto | RF | Esforço Estimado |
|---|---|---|---|
| Exportação de faturas para PDF (QuestPDF) | Alto — documento legal imprescindível | RF30 | 1–2 dias |
| Paginação de listagens (Stock, Vendas, Encomendas) | Médio — necessário com > 500 registos | RNF01 | 2 dias |
| Rastreamento de validades de produtos | Médio — gestão de desperdício | RF22 | 3 dias |
| Testes de integração com EF Core In-Memory | Alto — qualidade | — | 2 dias |
| Migração para ILogger + Serilog | Médio — monitorização | — | 1 dia |

## Propostas de Evolução — Versão 2.0 (Médio/Longo Prazo)

Para uma versão de produção real da QuickMart, seriam necessárias as seguintes extensões:

| Funcionalidade | Justificação |
|---|---|
| **Integração com AT (SAFT-PT)** | Cumprimento legal obrigatório para faturação real |
| **App mobile nativa** (React Native / MAUI) | Gerentes e funcionários preferem dispositivos móveis |
| **Sistema de pagamentos** (Multibanco, MBWAY) | POS sem pagamento digital é uma limitação crítica em produção |
| **Notificações push** (alertas de stock em tempo real) | Gerentes notificados instantaneamente sem ter de abrir o dashboard |
| **Previsão de procura com ML** | Encomendas sugeridas automaticamente com base no histórico |
| **Multi-empresa** (diferentes cadeias no mesmo sistema) | Transformar o SGCLC num produto SaaS para múltiplos clientes |
| **Auditoria completa** (change data capture) | Registo de todas as alterações a todas as entidades, não apenas stock |
