# 5.5 Requisitos de Armazenamento e Integridade

## Estimativa de Volume de Dados

Para um sistema de 5 lojas QuickMart com operação normal, as estimativas de volume de dados são as seguintes:

| Tabela | Registos/Dia | Registos/Mês | Registos/Ano |
|---|---|---|---|
| `Vendas` | ~435 (87/loja × 5 lojas) | ~13.000 | ~158.000 |
| `LinhasVenda` | ~2.175 (5 produtos/venda em média) | ~65.000 | ~793.000 |
| `Stocks` | Estático (≈ 2.500 registos: 500 produtos × 5 lojas) | — | — |
| `AjustesStock` | ~10 | ~300 | ~3.600 |
| `Consolidacoes` | 5 (uma/loja) | ~150 | ~1.825 |
| `Encomendas` | ~2 | ~60 | ~720 |
| `Faturas` | ~20 | ~600 | ~7.300 |

**Estimativa de crescimento anual da base de dados:** aproximadamente **50–80 MB** com SQLite (sem dados multimedia). Com SQL Server e índices completos, estima-se **150–200 MB/ano**.

## Garantias de Integridade

### Integridade Referencial

Todas as chaves estrangeiras são enforçadas na base de dados e ao nível do ORM (EF Core). O cenário de cascata é:

| Relação | Comportamento em cascata |
|---|---|
| `Vendas → Lojas` | RESTRICT — uma loja não pode ser eliminada se tiver vendas |
| `LinhasVenda → Vendas` | CASCADE DELETE — linhas eliminadas com a venda |
| `Stocks → Produtos` | RESTRICT — produto não pode ser eliminado se tiver stock |
| `Utilizadores → Lojas` | SET NULL — se uma loja for desactivada, o utilizador pode existir sem loja |

### Integridade Transaccional

O EF Core assegura que todas as operações de negócio críticas são executadas dentro de uma transacção atómica:

- **Registo de venda**: INSERT em `Vendas` + INSERT em `LinhasVenda` + UPDATE em `Stocks` — ou tudo é persistido ou nada é alterado
- **Receção de encomenda**: UPDATE do estado da encomenda + UPDATE do stock — atómico
- **Consolidação diária**: INSERT em `Consolidacoes` + cálculos agregados — dentro de uma transacção por loja

### Integridade de Domínio

As seguintes restrições de domínio são enforçadas ao nível da base de dados (CHECK constraints):

| Restrição | Tabela | Condição |
|---|---|---|
| Stock não pode ser negativo | `Stocks` | `Quantidade >= 0` (enforçado ao nível da aplicação) |
| Quantidade vendida positiva | `LinhasVenda` | `CHECK (Quantidade > 0)` |
| Preço unitário positivo | `LinhasVenda` | `CHECK (PrecoUnitario > 0)` |
| Desconto não negativo | `LinhasVenda` | `CHECK (Desconto >= 0)` |
| Quantidade encomendada positiva | `LinhasEncomenda` | `CHECK (QuantidadePedida > 0)` |

## Política de Retenção de Dados

Na v1.0, os dados são retidos indefinidamente (sem política de eliminação automática). Para versões futuras, recomenda-se:
- **Arquivo de vendas** com mais de 3 anos para tabela de histórico
- **Purga de sessões** expiradas (geridas em memória — sem persistência)
- **Rotação de logs de auditoria** (`AjustesStock`) após 5 anos
