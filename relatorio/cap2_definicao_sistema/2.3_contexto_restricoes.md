# 2.3 Análise de Contexto e Restrições

## Contexto de Operação

O SGCLC opera numa arquitectura cliente-servidor em que as lojas QuickMart acedem ao sistema através de um browser web sobre uma rede local ou internet. Existe um único servidor central que agrega os dados de todas as lojas — tanto para efeitos de operação em tempo real como para a consolidação diária automática.

```
[ Browser – Loja Braga ]    ┐
[ Browser – Loja Palmeira ] ├──► [ Servidor ASP.NET Core / Blazor ] ──► [ Base de Dados SQLite/SQL Server ]
[ Browser – Loja Barcelos ] ┘         ↑
                                  SignalR (WebSocket)
```

A comunicação entre o browser e o servidor ocorre através de **SignalR** (WebSocket), tecnologia nativa do Blazor Server, que mantém uma ligação persistente e permite actualizações de UI em tempo real sem necessidade de recarregamento de página.

## Restrições Técnicas

| ID | Restrição | Origem | Impacto no Design |
|---|---|---|---|
| RT-01 | A aplicação deve ser uma aplicação web, acessível por browser sem instalação de software adicional | Enunciado LI4 | Adoptou-se Blazor Server, eliminando a necessidade de cliente nativo |
| RT-02 | O backend deve ser desenvolvido em .NET (C#) | Enunciado LI4 | Stack ASP.NET Core 8 + EF Core 8 |
| RT-03 | A base de dados de desenvolvimento é SQLite | Decisão da equipa | Zero-configuration, portável; migração para SQL Server em produção |
| RT-04 | Não existe integração com sistemas de pagamento externos | SRS §1.2 – Fora de Âmbito | O POS regista vendas mas não processa pagamento eletrónico |
| RT-05 | Não existe módulo de contabilidade fiscal avançada | SRS §1.2 – Fora de Âmbito | A faturação cobre emissão e consulta mas não integração com AT (Autoridade Tributária) |
| RT-06 | Não existe módulo de RH | SRS §1.2 – Fora de Âmbito | A gestão de utilizadores cobre apenas criação de contas e atribuição de papéis |

## Restrições Organizacionais

| ID | Restrição | Origem |
|---|---|---|
| RO-01 | O sistema deve suportar 3 papéis de utilizador distintos com permissões não sobreponíveis: GestorCadeia, GerenteLoja, Funcionario | Requisito do cliente (Ata nº1) |
| RO-02 | Cada Gerente de Loja apenas tem visibilidade sobre os dados da sua própria loja | Requisito do cliente (Ata nº2) |
| RO-03 | O Gestor da Cadeia tem visibilidade global mas não opera o POS de nenhuma loja | Requisito do cliente (Ata nº2) |
| RO-04 | A consolidação diária deve ocorrer automaticamente sem intervenção humana | Requisito do cliente (Ata nº3) |
| RO-05 | O sistema deve manter um log de auditoria de todos os ajustes de stock | Requisito do cliente (Ata nº2) |

## Restrições de Segurança

| ID | Restrição |
|---|---|
| RS-01 | As passwords são armazenadas exclusivamente em formato hash BCrypt (custo ≥ 12) — nunca em texto claro |
| RS-02 | O acesso a cada módulo é controlado por um mecanismo de RBAC implementado no `SessionService` singleton |
| RS-03 | As sessões de utilizador expiram após 8 horas de inactividade |
| RS-04 | A aplicação deve ser servida em HTTPS em ambiente de produção |

## Pressupostos

- Os utilizadores têm acesso a um browser moderno (Chrome, Firefox, Edge, Safari — versões recentes)
- A infraestrutura de rede interna das lojas garante latência aceitável ao servidor central
- A equipa de desenvolvimento tem pleno acesso ao código e à base de dados durante todo o desenvolvimento
- O caso de estudo QuickMart é fictício; os dados utilizados são sintéticos e gerados para fins de demonstração
