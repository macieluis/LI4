# 4.7 Validação da Especificação Construída

## Processo de Validação da Especificação

Após a elaboração do SRS na sua versão final (v1.0), a especificação foi sujeita a um processo formal de validação com dois objectivos: verificar que a especificação é internamente correcta e consistente, e confirmar que representa fielmente as necessidades dos stakeholders.

## Revisão pelo Cliente

Na reunião de **20 de Fevereiro de 2026**, o cliente João Martins reviu o SRS v1.0 integralmente. O processo de revisão incluiu:
1. Leitura partilhada da lista de RFs (42) — o cliente confirmou ou clarificou cada um
2. Validação da matriz RBAC (quem acede ao quê)
3. Revisão das prioridades (Essencial/Importante/Útil) — o cliente reordenou 3 RFs
4. Confirmação do âmbito "fora do sistema" (pagamentos, contabilidade fiscal, RH)

**Resultado:** SRS v1.0 aprovado com 2 alterações: RF22 (validades) rebaixado para "Importante" e RF34 (retry de consolidação) elevado para "Essencial".

## Revisão Técnica pela Equipa

Cada membro da equipa reviu independentemente a especificação, focando-se na sua área de responsabilidade:

| Revisor | Foco | Observações |
|---|---|---|
| Responsável de Requisitos | Completude e conformidade IEEE 830 | Acrescentadas notas de esclarecimento em RF08 e RF27 |
| Responsável de Arquitetura | Viabilidade técnica de cada RF | Sinalizado RF30 (PDF) como "requer biblioteca externa" |
| Responsável de Implementação | Testabilidade de cada RF | Confirmado: todos os RFs são verificáveis por testes ou inspecção |
| Responsável de Testes | Critérios de aceitação | Adicionados critérios de aceitação explícitos em 8 RFs |

## Checklist de Qualidade SRS (IEEE 830)

| Critério | Resultado |
|---|---|
| Todos os RFs têm identificador único (RF01–RF42) | ✅ |
| Todos os RFs têm prioridade definida | ✅ |
| Todos os RFs têm estado de validação (Aprovado/Implementado) | ✅ |
| Todos os RNFs têm critério de verificação mensurável | ✅ |
| Matriz de RBAC cobre todos os módulos por todos os papéis | ✅ |
| Rastreabilidade RF → US → UC documentada | ✅ |
| Dependências entre RFs documentadas | ✅ (ex. RF27 depende de RF11) |
| Âmbito do sistema explicitamente definido (in/out) | ✅ |
| Glossário de termos incluído | ✅ (SRS §1.3) |

## Impacto na Fase de Design

A validação da especificação revelou dois casos em que o design teve de ser ajustado para acomodar os requisitos validados:

1. **RF22 (Validades)**: A decisão de o mover para v1.1 libertou o design da necessidade de uma tabela `Validades` na v1.0, simplificando o esquema da base de dados.

2. **RF09 (Pesquisa insensível a acentos)**: A validação confirmou este requisito, o que levou à decisão de implementar a normalização via `NormalizationForm.FormD` tanto no repositório (`Repositories.cs`) como no filtro client-side do POS (`Index.razor`), garantindo consistência em toda a aplicação.
