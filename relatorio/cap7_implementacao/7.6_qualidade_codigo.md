O código produzido foi revisto para verificar conformidade com a arquitectura em 3 camadas definida na Etapa 2. Os seguintes critérios foram verificados:

| Critério | Estado |
|---|---|
| Componentes Blazor não acedem directamente ao `AppDbContext` | ✅ — apenas via serviços |
| Serviços não importam namespaces de UI (`Blazor`, `Components`) | ✅ — separação completa |
| Repositórios não contêm lógica de negócio (apenas CRUD) | ✅ — lógica nos serviços |
| Todas as operações de BD são async (sem bloqueios síncronos) | ✅ — `await` em todo o acesso à BD |
| Nenhuma password em texto claro no código ou logs | ✅ — BCrypt em todo o lado |

## Decisões de Qualidade Relevantes

### DTOs vs Entidades de Domínio

A camada de apresentação nunca recebe directamente entidades EF Core — trabalha sempre com DTOs. Isto previne *over-fetching*, ciclos de serialização JSON infinitos e exposição de campos sensíveis:

```
Repositório → Entidade EF Core → Serviço → DTO → Componente Blazor
```

### Tratamento de Erros

Os erros de negócio são tratados com excepções específicas (`InvalidOperationException` com mensagem descritiva) que os componentes Blazor capturam e apresentam ao utilizador:

```csharp
// No componente Blazor
try {
    await StockService.AjustarStockAsync(...);
    mensagemSucesso = "Stock actualizado com sucesso.";
} catch (InvalidOperationException ex) {
    mensagemErro = ex.Message; // ex: "Stock não pode ficar negativo"
}
```

### Async/Await Completo

Todas as operações de I/O (acesso à BD, operações assíncronas) utilizam correctamente `async/await`, sem *blocking calls* (`.Result`, `.Wait()`), garantindo que o servidor Blazor não fica bloqueado durante operações de BD.
