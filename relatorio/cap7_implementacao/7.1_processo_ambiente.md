# 7.1 Processo de Implementação e Ambiente de Desenvolvimento

## Abordagem à Implementação

A implementação do SGCLC seguiu o plano definido na Etapa 2, respeitando a arquitectura em 3 camadas e os padrões de design estabelecidos (Repository, DI, DTO). O desenvolvimento foi incremental por módulo, seguindo a prioridade dos RFs: Autenticação → Produtos → Stock → POS → Encomendas → Faturação → Consolidação → Relatórios.

Cada módulo foi desenvolvido nas três camadas em sequência:
1. **Core** — entidade de domínio + DTO + interface de serviço
2. **Data** — repositório (implementa acesso à BD via EF Core)
3. **Web** — serviço de aplicação + componente Blazor

## Setup do Projecto

O projecto .NET foi inicializado com três projects na mesma solução:

```bash
dotnet new sln -n ConvenienceChain
dotnet new classlib -n ConvenienceChain.Core -f net8.0
dotnet new classlib -n ConvenienceChain.Data -f net8.0
dotnet new blazorserver -n ConvenienceChain.Web -f net8.0

dotnet sln add src/ConvenienceChain.Core/
dotnet sln add src/ConvenienceChain.Data/
dotnet sln add src/ConvenienceChain.Web/
```

## Configuração da Aplicação

O ficheiro `appsettings.json` define a connection string da base de dados:

```json
{
  "ConnectionStrings": {
    "DefaultConnection": "Data Source=sgclc.db"
  },
  "Logging": {
    "LogLevel": {
      "Default": "Information"
    }
  }
}
```

O `launchSettings.json` (criado para garantir porta consistente) define:

```json
{
  "profiles": {
    "ConvenienceChain.Web": {
      "commandName": "Project",
      "applicationUrl": "http://localhost:5050",
      "environmentVariables": {
        "ASPNETCORE_ENVIRONMENT": "Development"
      }
    }
  }
}
```

## Decisões de Implementação Relevantes

| Decisão | Alternativa Considerada | Razão da Escolha |
|---|---|---|
| Blazor Server | Blazor WebAssembly, MVC | Server-side rendering via SignalR; sem necessidade de API REST separada; acesso directo aos serviços |
| SQLite para dev | SQL Server Express | Zero-configuration; portabilidade entre macOS e Windows; troca por SQL Server em prod via connection string |
| GUID como PK em Utilizadores | INT AUTO_INCREMENT | Evita colisões em cenários de migração em larga escala; prática comum em sistemas de autenticação |
| Nenhum HTTPS em dev | HTTPS sempre activo | Evita problemas de certificado auto-assinado; HTTPS configurado para produção |
