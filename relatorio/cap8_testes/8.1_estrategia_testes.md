# 8.1 Estratégia de Testes

## Abordagem à Verificação e Validação

A estratégia de testes do SGCLC foi definida na Etapa 4 e baseia-se na pirâmide de testes clássica, adaptada às características do sistema:

```
              /\
             /  \
            / UI \         ← Testes manuais (fluxos completos)
           /──────\
          / Integr.\      ← Não implementados explicitamente (EF Core In-Memory seria v1.1)
         /──────────\
        /  Unitários \    ← 10 testes xUnit (classes de serviço)
       /______________\
```

A escolha de focar os testes automatizados nos **serviços de negócio** (e não nos repositórios EF Core nem nos componentes Blazor) segue o princípio de que a lógica de negócio é a área de maior risco e maior valor para testar.

## Âmbito de Testes

| Tipo de Teste | Cobertura | Ferramentas |
|---|---|---|
| **Testes Unitários** | StockService (5 testes), SalesService (3 testes), ProdutoService (2 testes) | xUnit 2.7, Moq 4.20, FluentAssertions 6.12 |
| **Testes Manuais** | 9 fluxos principais de utilizador | Execução manual na aplicação |
| **Testes de Aceitação** | 5 critérios de interface (secção 6.4) | Execução manual + verificação visual |
| **Testes de Qualidade** | ISO/IEC 25010 — 7 características | Avaliação por inspecção e medição |

## Ambiente de Testes

Os testes unitários são executados isoladamente, sem acesso à base de dados real. Os repositórios são mocados com **Moq**, garantindo que os testes são rápidos, determinísticos e não dependent da BD:

```bash
cd src/ConvenienceChain.Tests/
dotnet test --verbosity normal
```

## Critérios de Aceitação de Testes

Os seguintes critérios devem ser satisfeitos para a release da v1.0:

| Critério | Limiar | Resultado |
|---|---|---|
| Testes unitários passados | 10/10 (100%) | ✅ 10/10 |
| Build sem erros | 0 erros | ✅ 0 erros |
| Fluxos manuais críticos verificados | 9/9 | ✅ 9/9 |
| Critérios de aceitação de interface | 5/5 | ✅ 5/5 |
| Cobertura de RFs implementados | ≥ 90% | ✅ 90.5% (38/42) |
